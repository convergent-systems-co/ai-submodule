# JM Naming Convention Adoption Guide

## Overview

All Azure resources deployed via Bicep in JM Family repositories must use deterministic names generated by the centralized `getResourceNames` utility module. This guide walks through adopting the naming convention in a consuming repository.

## Prerequisites

- Azure Container Registry access configured (see `bicepconfig.json` setup below)
- Bicep CLI >= 0.28 installed
- Repository bootstrapped with `.ai/bin/init.sh`

## Step 1: Configure bicepconfig.json

Ensure your project root contains a `bicepconfig.json` with the ACR module alias:

```json
{
  "moduleAliases": {
    "br": {
      "acr-prod": { "registry": "crtoprodbicepregistrya.azurecr.io" },
      "acr-dev": { "registry": "crtodevbicepregistrya.azurecr.io" }
    }
  }
}
```

## Step 2: Import getResourceNames

In every Bicep file that creates named Azure resources, add the import at the top:

```bicep
import { getResourceNames } from 'br/acr-prod:modules/util:v0'
```

Define the four required parameters that feed the naming function:

```bicep
param businessUnit string      // e.g., 'set', 'jma', 'jmf'
param deploymentStage string   // e.g., 'dev', 'stg', 'uat', 'prod'
param applicationName string   // e.g., 'bankingapp'
param applicationId string     // e.g., 'b' (single letter a-z)
```

Call the function and store the result:

```bicep
var names = getResourceNames(businessUnit, deploymentStage, applicationName, applicationId)
```

## Step 3: Use Generated Names for All Resources

Replace any hardcoded or interpolated resource names with properties from the `names` output:

```bicep
// Key Vault (mini pattern: kvsetdevbankingapp)
module kv 'br/acr-prod:modules/core/keyvault:v4' = {
```

## Step 4: Update Destroy/Purge Pipelines

Destroy and purge pipelines must reference the same naming convention to correctly target soft-deleted resources. Import `getResourceNames` in your destroy Bicep or use the same names in pipeline scripts:

```yaml
# In your destroy/purge workflow
steps:
  - name: Purge soft-deleted Key Vault
    run: |
      # Use the same naming convention as deployment
      KV_NAME="kv${BU}${STAGE}${APP_NAME}"
      az keyvault purge --name "$KV_NAME" --no-wait

  - name: Purge soft-deleted SQL Server
    run: |
      SQL_NAME="sql-${BU}-${STAGE}-${APP_NAME}-${APP_ID}"
      az sql server delete --name "$SQL_NAME" --resource-group "rg-${BU}-${STAGE}-${APP_NAME}-${APP_ID}" --yes
```

The preferred approach is to use the same `getResourceNames` import in a dedicated destroy Bicep file so the names stay in sync automatically:

```bicep
import { getResourceNames } from 'br/acr-prod:modules/util:v0'

param businessUnit string
param deploymentStage string
param applicationName string
param applicationId string

var names = getResourceNames(businessUnit, deploymentStage, applicationName, applicationId)

// Output names for pipeline scripts to consume
output keyVaultName string = names.keyVault
output sqlServerName string = names.sqlServer
output storageAccountName string = names.storageAccount
```

## Step 5: Update project.yaml

Add the naming convention configuration to your `project.yaml`:

```yaml
conventions:
  naming:
    require_util_module: true
    util_module: "br/acr-prod:modules/util:v0"
    import_statement: "import { getResourceNames } from 'br/acr-prod:modules/util:v0'"
    patterns:
      standard: "{prefix}-{bu}-{stage}-{appName}-{appId}"
      mini: "{prefix}{bu}{stage}{product}"
      small: "{prefix}-{bu}-{stage}-{product}"
```

## Naming Patterns Reference

The util module applies three naming patterns based on Azure resource type constraints:

### Standard Pattern

**Format:** `{prefix}-{bu}-{stage}-{appName}-{appId}`

Used by most resources with no strict character limits.

| Resource | Prefix | Example (banking-app, dev, appId=b) |
|----------|--------|-------------------------------------|
| Resource Group | `rg-` | `rg-set-dev-bankingapp-b` |
| SQL Server | `sql-` | `sql-set-dev-bankingapp-b` |
| SQL Database | `sqldb-` | `sqldb-set-dev-bankingapp-b` |
| Managed Identity | `id-` | `id-set-dev-bankingapp-b` |
| App Service Plan | `asp-` | `asp-set-dev-bankingapp-b` |
| App Service | `app-` | `app-set-dev-bankingapp-b` |
| Function App | `func-` | `func-set-dev-bankingapp-b` |
| Logic App | `logic-` | `logic-set-dev-bankingapp-b` |
| API Management | `apim-` | `apim-set-dev-bankingapp-b` |
| Cosmos DB | `cosmos-` | `cosmos-set-dev-bankingapp-b` |
| Redis Cache | `redis-` | `redis-set-dev-bankingapp-b` |
| Service Bus NS | `sbns-` | `sbns-set-dev-bankingapp-b` |
| Event Hub NS | `evhns-` | `evhns-set-dev-bankingapp-b` |
| Event Grid Topic | `evgt-` | `evgt-set-dev-bankingapp-b` |
| Front Door Profile | `afd-` | `afd-set-dev-bankingapp-b` |
| Public IP | `pip-` | `pip-set-dev-bankingapp-b` |
| Synapse Workspace | `synw-` | `synw-set-dev-bankingapp-b` |
| Integration Account | `ia-` | `ia-set-dev-bankingapp-b` |
| VNet | `vnet-` | `vnet-set-dev-bankingapp-b` |
| Subnet | `snet-` | `snet-set-dev-bankingapp-b` |

### Mini Pattern

**Format:** `{prefix}{bu}{stage}{product}` (no hyphens, 24-character limit)

Used by resources with strict character limits that prohibit hyphens.

| Resource | Prefix | Example (banking-app, dev) |
|----------|--------|----------------------------|
| Key Vault | `kv` | `kvsetdevbankingapp` |
| Storage Account | `st` | `stsetdevbankingapp` |
| Container Registry | `cr` | `crsetdevbankingapp` |

### Small Pattern

**Format:** `{prefix}-{bu}-{stage}-{product}` (60-character limit)

Used by resources with moderate character limits where hyphens are allowed but the application ID is omitted.

| Resource | Prefix | Example (banking-app, dev) |
|----------|--------|----------------------------|
| App Configuration | `appcs-` | `appcs-set-dev-bankingapp` |
| App Insights | `appi-` | `appi-set-dev-bankingapp` |
| Log Analytics | `log-` | `log-set-dev-bankingapp` |

## Business Units

Valid business unit values: `jma`, `jmf`, `jmfe`, `set`, `setf`, `to` (TechOps), `ocio`, `octo`, `lexus`

## Deployment Stages

Valid deployment stage values: `dev`, `stg`, `uat`, `prod`, `nonprod`

## Common Mistakes

| Mistake | Correct Approach |
|---------|-----------------|
| Hardcoding resource names as string literals | Use `names.{resourceType}` from `getResourceNames()` output |
| Using string interpolation like `'kv${bu}${stage}${app}'` | Use `names.keyVault` — the util module handles format rules |
| Different names in deploy vs. destroy pipelines | Import `getResourceNames` in both deploy and destroy Bicep files |
| Omitting the import in Bicep files that reference resources by name | Every file that uses a resource name must import the util module |
| Using `acr-dev` registry alias for production deployments | Always use `acr-prod` for the util module import |

## Validation

The architecture review panel includes naming convention compliance checks for Bicep changes. The Infrastructure Engineer perspective verifies:

1. `getResourceNames` import is present in all Bicep files that create named resources
2. Resource names are derived from the function output, not hardcoded
3. Destroy/purge pipelines use the same naming source
4. No naming drift between deployment artifacts
